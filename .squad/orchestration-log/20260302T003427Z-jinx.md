### Key Architectural Decisions

1. **External scripts over built-in auto-update**
   - Cleaner separation of concerns
   - Graphics application doesn't manage system updates
   - Scheduled via cron/Task Scheduler without modifying core code

2. **Dual-script approach (bash + PowerShell)**
   - Platform-specific process handling (Windows file locking vs. Linux inode swap)
   - Native tools, no Python dependency for update process
   - Follows existing install.sh/install.ps1 convention

3. **5-Phase Update Workflow**
   - Check: Compare local VERSION against GitHub API latest release
   - Download: Fetch ZIP (gh CLI preferred, direct URL fallback)
   - Stop: Gracefully stop pumpkin_face.py with force-kill fallback
   - Deploy: Extract, validate, copy files, run pip install
   - Restart: Launch with original command line arguments

4. **Process Detection via Command Line Pattern**
   - Linux/macOS: `pgrep -f "python.*pumpkin_face.py"`
   - Windows: WMI query `Get-WmiObject Win32_Process`
   - Capture full command arguments for restart preservation

5. **Semantic Version Comparison**
   - Compare major.minor.patch as integer tuples
   - Strip 'v' prefix from tags
   - Pre-release versions documented as v1 limitation

6. **Idempotent & Safe**
   - Multiple runs produce same result
   - Logs "Already up-to-date" and exits 0 if no update available
   - ZIP validation before deployment

## Insights & Learnings

**Architectural Pattern:** External updater scripts provide clean separation of concerns. Graphics application remains focused on rendering; system administration logic lives in scheduled scripts.

**Process Handling Complexity:** Windows and Linux have fundamentally different process management models. Native scripts are simpler and more reliable than cross-platform abstraction.

**Idempotency as Core Design:** Update script must be safe to run from cron jobs. Idempotent design eliminates timing race conditions.
